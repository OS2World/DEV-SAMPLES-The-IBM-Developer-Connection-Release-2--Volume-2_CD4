REXX to CICS Interface for OS/2 (M1)

This product is located on Member Disc 1.

CICS for OS/2 provides an excellent capability to connect personal work stations to CICS hosts thereby allowing workstation programs to link to host programs and to access host data through the distributed program link (DPL). However, traditional CICS for OS/2 programs are bound to a 3270-terminal-type interface i.e. they are not able to directly use the graphical features available on an OS/2 workstation.

The External Call Interface (ECI) provided with CICS for OS/2 and CICS Client for OS/2 and the External Transaction Initiation (ETI) provided with CICS for OS/2 allow access to CICS facilities and data from non-CICS applications running either under OS/2 in the same workstation as the CICS for OS/2 system or (for ECI) on a client in a CICS for OS/2 client/server configuration. It is therefore possible to develop graphical user interface (GUI) front ends for CICS applications using OS/2 Presentation Manager or other presentation systems.

Before the functionality provided by the ECI facility can be used it is necessary to write a native OS/2 application for this pupose. This can involve a certain amount of learning time depending upon the experience of the programmer and the tools used to build the application.

REXX, on the other hand, is a language which allows for easy and highly efficient programming. A particularly useful feature is the ability to execute any OS/2 program as an external command. This allows for the fast prototyping of relatively complex applications within a short period of time. In addition, the native function set of REXX can be extended through the concept of external functions. Thus, even though REXX has no native PM interface, there are several products and tools available (Hockware's VISPRO/REXX, Watcoms Visual REXX, VREXX2 (an EWS package) and DRREXX from the DRIALOG package on OS2TOOLS) which extend the REXX language to include PM programming features.

Prior to CICS Client for OS/2 V1.1, however, there was no REXX interface to ECI for OS/2 and there remains no REXX interface to the ETI. The ability to provide the combination of REXX programming with easy access to host data would therefore be an attractive feature in many network environments consisting of personal workstations connected to CICS hosts.

This SupportPac provides this capability via a Dynamic Link Library (RxECI) which can be accessed from REXX as an external function. This DLL can be used from both a CICS OS/2 client and a CICS for OS/2 server. The additional ETI interface feature can be used from a server only, and is provided in the DLL RxETI. A prototype DLL (RXEPI.DLL) for the EPI interface is also provided.

Systems Specialists providing the service of prototyping an application using RxCICS should be skilled to specialist level in CICS - CICS for OS/2 and be familiar with REXX programming.

Further Details on CA31:

It is envisaged that RxCICS can be used to undertake fast prototyping of an application to demonstrate how CICS for OS/2 can be used to provide a solution to a particular business problem.

  • RxCICS requires OS/2 REXX to be installed in addition to CICS for OS/2 Version 2.0 or later or CICS Client for OS/2.
  • When installed on a CICS for OS/2 Version 2.0.1 system, Corrective Service Disk 5 must be installed for REXX ECI calls to work correctly on the CICS for OS/2 Server (REXX ECI calls from a CICS for OS/2 distributed client or CICS Client for OS/2 are not affected).
  • A REXX ECI interface, with samples and based on this SupportPac, was introduced into the CICS Client for OS/2 at Version 1.1 and later. The calling convention is different between the two versions and thus you will need to modify both sets of samples if you want to use them in conjunction with the other ECI interface.
  • The ETI interface was not ported but will work with either CICS for OS/2 V2 or Transaction Server for OS/2.
